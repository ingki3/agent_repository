# Agent Client — PRD

## 1. Vision & Problem
   1.1 Vision
   Agent Client는 **에이전트와의 연결을 위한 전용 메신저**다. 사용자는 메신저의 친구를 추가하듯 여러 에이전트(예: 회사일 지원 에이전트, 개인 일상 지원 에이전트)를 각각의 "버디(Buddy)"로 등록하고, Telegram 호환 프로토콜 위에서 익숙한 채팅 UX로 대화하며 이메일 확인·정보 정리·일정/주소록 관리·쇼핑·여행 계획 등 일상 업무를 위임할 수 있다. 동시에 일반 메신저에는 없는 에이전트 특화 UI — 인라인 액션, 카드, 그리고 에이전트가 "생각하고(thinking)" "도구를 호출(tool calling)" 하는 과정의 가시화 — 를 모바일 네이티브로 제공한다.

   1.2 Problem statement
   - 통점 1 (앱 파편화): 이메일, 캘린더, 쇼핑, 지도, 메모 등 서로 다른 앱을 오가며 정보를 모으고 정리하는 비용이 크다.
   - 통점 2 (기존 Telegram 봇의 한계): 일반 Telegram 앱은 사람 간 메시징 중심이라 에이전트 특화 UI(확인 버튼, 카드, 폼)와 상태 표시, 추론·도구 호출 과정의 가시화를 제대로 지원하지 못한다.
   - 통점 3 (Telegram 마크다운 가독성 문제): Telegram은 LLM이 흔히 생성하는 마크다운(헤더 `#`, 표, 중첩 리스트, 코드 펜스, 인라인 링크 등)을 제대로 렌더링하지 않고 원본 텍스트(`**bold**`, `### 제목`)를 그대로 노출하는 경우가 많아 모바일에서 읽기가 매우 불편하다. 에이전트 응답은 대개 구조화된 마크다운이라 이 격차는 일상 사용 시 즉시 체감된다.
   - 통점 4 (다중 에이전트 파편화): 여러 에이전트를 각기 다른 봇/앱으로 운영하면 단일 통합 뷰(Inbox)가 없어 상태 파악과 컨텍스트 스위칭 비용이 크다. 회사·개인 등 역할별 에이전트를 깔끔하게 분리·관리할 수단도 없다.
   - 통점 5 (블랙박스 에이전트): 에이전트가 결과만 던지면 사용자는 "왜 그렇게 답했는지" "어떤 데이터를 봤는지" 신뢰하기 어렵다. thinking·tool call·결과를 단계적으로 볼 수 있어야 안심하고 위임할 수 있다.
   - 통점 6 (모바일 사용성 부재): 기존 에이전트 웹 콘솔은 데스크톱에 최적화되어, 이동 중이나 짧은 자투리 시간에 즉시 정보를 확인하거나 작업을 위임하기 어렵다.
   - 기존 해결책의 한계: 일반 Telegram은 에이전트 특화 인터랙션 확장이 막혀 있고, 웹 UI는 모바일 접근성·알림성이 떨어진다.

   1.3 Why now
   LLM 에이전트가 이메일/캘린더/쇼핑/여행 등 일상 도메인 툴(MCP, OAuth 연동)과 결합되면서, 한 번의 채팅으로 "정보 수집 → 정리 → 행동"을 한 번에 처리하는 시대가 열렸다. 익숙한 Telegram UX와 봇 프로토콜을 차용해 진입 장벽을 낮추면서도, 다수의 에이전트를 버디로 모아놓고 추론 과정까지 들여다볼 수 있는 모바일 네이티브 전용 메신저가 필요한 시점이다.

## 2. Target Users & Personas
   2.1 페르소나 1~3명

   **페르소나 A: 바쁜 직장인 (Daily Planner)**
   - 역할/숙련도: 30대 직장인. 메신저·캘린더·쇼핑 앱을 일상적으로 쓰지만 AI 에이전트는 초·중급 수준.
   - 목적: 출퇴근길에 이메일·일정을 빠르게 정리하고, 쇼핑/예약 같은 잡무를 채팅으로 위임. 회사용 에이전트와 개인용 에이전트를 분리해 관리.
   - 등록 버디 예시: "Work Buddy"(사내 이메일·캘린더·이슈트래커 연동 에이전트), "Life Buddy"(개인 Gmail·쇼핑·여행 에이전트). 두 버디를 별도 채널로 두고 통합 Inbox에서 함께 확인.
   - 대상 에이전트: SimpleClaw, OpenClaw (Telegram 호환 브릿지) — 이메일·캘린더·연락처·쇼핑 툴 연동.

   **페르소나 B: 여행·취미 플래너 (Trip Planner)**
   - 역할/숙련도: 20~40대 사용자. 여행/외식/문화생활을 자주 계획하고, 정보 검색과 비교에 시간을 많이 쓴다.
   - 목적: 자연어 한 줄로 여행 일정 초안, 항공·숙소 후보, 식당 리스트를 받아보고 캘린더에 바로 등록.
   - 대상 에이전트: 여행/검색/지도 스킬이 활성화된 SimpleClaw 채널.

   **페르소나 C: 정보 큐레이터 (Knowledge Keeper)**
   - 역할/숙련도: 학생·전문직 등 매일 다량의 뉴스·문서·링크를 소비하는 사용자.
   - 목적: 본 링크/문서/메시지를 에이전트에게 보내 요약·태깅·아카이브하고, 나중에 자연어로 검색.
   - 대상 에이전트: 요약·메모·검색 스킬을 가진 OpenClaw 채널.

   2.2 페르소나별 핵심 작업 흐름
   - 일상 흐름: 아침 출근길/점심시간/잠들기 전 등 자투리 시간에 모바일로 에이전트에 말 걸기.
   - 트리거: 새 이메일/일정 알림, 떠오른 할 일, 쇼핑 결정, 여행/외출 준비, 정리하고 싶은 정보.
   - 핵심 작업: 통합 Inbox에서 에이전트들의 브리핑·제안을 확인하고, 카드/버튼으로 원터치 확정(예약/구매/일정 추가), 텍스트·음성으로 자연스럽게 후속 지시.
   - 단일 사용자 시점: 1차 PRD 범위에서는 단일 사용자(Single User, 개인용)로 한정. 가족 공유, 다중 사용자 권한 모델은 핵심 가치 구현 후 검토.

   3.2 핵심 UC 3개 상세 (User Journey)

   **UC-01: 아침 출근길 이메일·일정 모닝 브리핑**
   - 트리거: 사용자가 출근길에 앱을 열거나, 매일 아침 8시 에이전트가 자동 브리핑 카드를 푸시.
   - 단계별 흐름:
     1. 사용자: 알림 탭 또는 앱 진입 후 마이크 버튼을 눌러 "오늘 일정이랑 중요한 메일만 알려줘"라고 발화.
     2. 시스템: STT 변환 후 OpenClaw로 전송. 에이전트가 메일/캘린더 툴을 호출.
     3. 시스템: "오늘 일정 3건, 답장이 필요한 이메일 2건이 있어요." 텍스트 + TTS 응답, 하단에 [일정 카드] [메일 카드] 인라인 카드 렌더링.
     4. 사용자: '메일 카드'의 [요약 보기] 탭. 발신자/제목/3줄 요약 표시. [답장 초안 만들기] 버튼 클릭.
     5. 시스템: 답장 초안을 채팅에 표시. 사용자가 음성으로 "마지막 줄 부드럽게 바꿔줘" 수정 후 [전송] 확정.
   - 성공 종료 조건: 사용자가 메일 1건 답장 발송, 일정 1건을 머릿속에 정리한 상태로 출근.
   - 실패/대안 경로: OAuth 토큰 만료 시 인라인 카드로 재인증 버튼 노출. 메일 본문이 길면 [더 보기]로 단계적 확장.
   - 사용된 에이전트 capability: Gmail/IMAP 스킬, Calendar 스킬, 요약 레시피, STT/TTS.

   **UC-05: 주말 여행 일정 초안 만들기**
   - 트리거: 사용자가 채팅창에 "다음 주 토요일에 부산 1박 2일 여행 계획 짜줘. 바다 보이는 숙소 선호."
   - 단계별 흐름:
     1. 시스템: SimpleClaw가 의도 파악 후 날짜/예산/인원에 대한 확인 질문을 1회 던짐 ("2명 기준 1박 20만원 이내로 보면 될까요?").
     2. 사용자: "응, 2명이고 25만원까지."
     3. 시스템: 여행/지도/검색 스킬을 호출해 일정 카드 생성 — "토요일 오전 KTX → 해운대 체크인 → 점심 식당 후보 3곳 → 저녁 야경 코스".
     4. 시스템: 숙소 후보 카드 3개(이미지 썸네일, 가격, 위치)와 [캘린더에 추가] [숙소 예약 페이지 열기] 인라인 버튼 렌더링.
     5. 사용자: 숙소 1번 선택 → [캘린더에 추가] 클릭.
     6. 시스템: Google Calendar에 일정 추가 후 "토요일 09:00 부산행 일정이 캘린더에 추가됐어요" 응답.
   - 성공 종료 조건: 여행 일정 초안이 캘린더에 등록되고, 숙소/식당 후보 정보가 채팅 스레드로 남음.
   - 실패/대안 경로: 검색 결과가 부족하면 에이전트가 [조건 완화] [다시 검색] 버튼 제공. 사용자가 "다른 도시도 후보로 보여줘"로 분기 가능.
   - 사용된 에이전트 capability: 여행/지도 검색 스킬, Calendar 쓰기 스킬, 카드 렌더링 메타데이터.

   **UC-07: 링크/뉴스 요약 및 아카이브**
   - 트리거: 사용자가 다른 앱(브라우저/카톡 등)에서 본 뉴스 링크를 OS 공유 시트를 통해 Agent Client로 전송.
   - 단계별 흐름:
     1. 사용자: 공유 시트에서 'OpenClaw에 보내기' 선택. 링크와 함께 "이거 요약하고 'AI 트렌드'로 태깅해줘" 입력.
     2. 시스템: 에이전트가 페이지 본문을 가져와 3줄 요약 + 핵심 키워드 추출. 채팅에 요약 카드 표시.
     3. 시스템: [내 메모에 저장] [태그 수정] [원문 열기] 버튼 렌더링.
     4. 사용자: 며칠 후 "지난주에 본 AI 트렌드 기사 뭐였지?"라고 검색.
     5. 시스템: 저장된 항목을 자연어 검색으로 찾아 카드 리스트로 표시.
   - 성공 종료 조건: 링크가 요약·태깅된 상태로 영구 저장되고, 나중에 자연어 질의로 다시 찾을 수 있음.
   - 실패/대안 경로: 페이지 크롤링 실패 시 사용자에게 본문 붙여넣기 요청. 태그 충돌 시 추천 태그 제시.
   - 사용된 에이전트 capability: 웹 페치/요약 스킬, 메모/노트 저장 스킬, 시맨틱 검색.

   **UC-10: 신규 에이전트(버디) 추가 및 채널 분리**
   - 트리거: 사용자가 회사에서 새로 발급받은 사내 에이전트(Work Buddy)를 추가하고 싶음.
   - 단계별 흐름:
     1. 사용자: 상단 [+] 버튼 → "버디 추가" 진입. 텔레그램과 동일하게 봇 토큰 입력 또는 봇 핸들/딥링크(`agent://add?token=...`)를 통해 등록.
     2. 시스템: 텔레그램 호환 Bot API로 `getMe` 호출, 봇 이름/아이콘/설명을 미리보기 카드로 표시.
     3. 사용자: 표시명을 "Work Buddy"로 지정, 알림 정책(업무시간만 등) 설정 후 [추가] 확정.
     4. 시스템: 채널 목록에 "Work Buddy"가 신규 버디로 등록되고, 환영 메시지(`/start`)가 자동 전송된다.
     5. 사용자: 기존 "Life Buddy"와 별개 채널로 운용. 통합 Inbox에서는 두 버디의 메시지가 시간순으로 함께 보이지만 채널 색상/뱃지로 출처를 구분.
   - 성공 종료 조건: 신규 버디가 채널 목록에 추가되고 메시지 송수신 동작.
   - 실패/대안 경로: 토큰 오류 시 사유 표시 및 재입력. 동일 봇이 이미 등록된 경우 중복 등록 방지 안내.
   - 사용된 에이전트 capability: Telegram 호환 Bot API (`getMe`, `setWebhook`, `sendMessage`), 채널 메타데이터 관리.

   **UC-11: 에이전트 thinking·tool call 과정 확인**
   - 트리거: 사용자가 UC-05 여행 계획 응답을 받았는데, 추천된 숙소가 정말 조건에 맞는지 의심.
   - 단계별 흐름:
     1. 시스템: 응답 메시지 하단에 접힌 형태의 "Working trace" 패널을 함께 표시한다. 기본 상태는 접혀 있고 "🧠 3단계로 사고함 · 🛠 5개 툴 호출" 같은 요약만 노출.
     2. 사용자: 패널을 펼친다.
     3. 시스템: 단계별로 다음을 렌더링한다.
        - **Thinking 단계**: 에이전트가 자가 추론한 plan/요약(예: "예산·바다뷰 조건 → 해운대 권역으로 좁히기").
        - **Tool calls**: 각 호출의 이름, 인자(JSON, 민감값 마스킹), 응답 시간, 결과 요약(예: `hotel_search({region:"해운대", max_price:250000}) → 12 hits`).
        - **Streaming 진행 상태**: 응답이 생성 중일 때 thinking·tool 노드가 실시간으로 추가되고 스피너가 표시됨.
     4. 사용자: 특정 tool call의 [원본 응답] 보기를 탭해 JSON 펼쳐보기, 또는 [이 단계 다시 실행]으로 같은 도구를 다른 인자로 재호출 요청.
     5. 시스템: 사용자가 패널을 다시 접으면 깔끔한 채팅 뷰로 복귀.
   - 성공 종료 조건: 사용자가 추천 근거(어떤 검색을 했고 어떤 결과를 봤는지)를 확인하고 결과를 신뢰하게 됨.
   - 실패/대안 경로: 일부 단계 트레이스가 누락된 경우 "이 단계 정보 없음"으로 회색 표시. 매우 긴 tool 응답은 처음 N줄만 표시 후 [전체 보기].
   - 사용된 에이전트 capability: 에이전트 게이트웨이의 thinking/tool-call 이벤트 스트림(SSE/WS), 트레이스 메타데이터.

   3.3 Out-of-scope use cases
   - 일반 지인과의 개인 메시징(P2P 채팅): 에이전트와의 소통에 특화.
   - 모바일 내 풀-피처 이메일·캘린더 편집기: 인라인 카드 수준의 가벼운 상호작용만 지원하고, 복잡한 편집은 원 앱에 위임.
   - 금융 결제 직접 수행: 쇼핑/예약은 후보 추천과 외부 결제 페이지 딥링크까지만 지원.
   - 다중 사용자 공유 스레드: 가족·팀이 한 채널을 공동 사용하는 그룹 챗 기능은 1차 제외.
   - 복잡한 차트/대시보드: 단순 요약 카드만 지원.

   4.2 Non-functional Requirements
   - 성능: 앱 콜드 스타트 2초 이내, 로컬 렌더링 지연 100ms 이내.
   - 보안/프라이버시: 외부 서비스 OAuth 토큰과 API Key는 OS 네이티브 Secure Storage에 저장, 통신은 HTTPS로 암호화. 이메일·연락처 등 개인정보는 디바이스 외부에 영구 저장하지 않는 것을 원칙으로 한다. 앱 자체에도 패스코드/생체 잠금 옵션을 제공해 분실·도난 시 보호.
   - 접근성: 스크린 리더 지원 및 명도 대비 WCAG AA 충족, 음성 입출력 우선 지원.
   - 국제화: 한국어 최우선, LTR 방향 적용. (영어는 후속)
   - 안정성: 네트워크 불안정 감지 및 크래시 복구(직전 채팅방 진입) 지원.
   - 관측성: 크래시 리포트 도구 연동, 개인정보 로그 마스킹.
   - 호환성: iOS 16+, Android 12+ 지원.

   4.3 Constraints & Assumptions
   - 기술 제약: 1차 릴리스는 단일 기기 기반의 로컬 저장소를 우선 활용한다.
   - 비즈니스 제약: 크로스 플랫폼 프레임워크(React Native, Expo)를 사용한다.
   - 외부 의존성: 에이전트 백엔드 측에 텔레그램 호환 브릿지가 안정적으로 구현되어 있고, 이메일·캘린더·쇼핑·지도 등 도메인 툴 스킬이 갖춰져 있다고 가정한다.
   - 사용자 가정: 사용자는 Gmail/Google Calendar 등 주요 외부 서비스 계정을 보유하며 OAuth 연동에 동의한다.

## 5. Technical Architecture
   5.1 시스템 컨텍스트 다이어그램
   ```
   [User]
     ↓ voice/text/share-sheet (OS Native STT/TTS, Share Extension)
   [Agent Client (Mobile App)]
     ↓ HTTP (REST) / SSE (Streaming) / WebSocket / FCM/APNs (Push)
   [Agent Gateway / Bot API Bridge]
     ↓
   [SimpleClaw / OpenClaw Agent Daemon] ←→ [LLM Routers (Claude/Gemini/OpenAI)]
                                        ←→ [Domain Skills]
                                            - Email (Gmail/IMAP)
                                            - Calendar (Google/Apple)
                                            - Contacts (People API)
                                            - Shopping/Search
                                            - Maps & Travel
                                            - Web fetch & Summarize
                                            - Notes/Memory & Semantic Search
   ```

   5.2 기술 스택 결정
   - 언어 / 프레임워크: React Native (Expo) - 크로스 플랫폼 개발 및 네이티브 모듈 연동.
   - 상태 관리: Zustand.
   - 음성 입출력: expo-speech (TTS), @react-native-voice/voice (STT).
   - 인증/보안: API Token + Expo SecureStore, OAuth 콜백은 에이전트 게이트웨이가 중개.
   - 저장: SQLite (expo-sqlite) - 오프라인 캐시 및 컨텍스트 복구.
   - 공유 입력: iOS/Android Share Extension (Expo Modules).
   - 빌드/배포: EAS (Expo Application Services).

   5.3 에이전트 연동 인터페이스
   - **Telegram 호환 Bot API 표준 지원**: 봇 등록·메시지 송수신·인라인 키보드·명령(`/start`, `/help` 등)을 텔레그램 봇 프로토콜 그대로 구현. 기존 텔레그램 봇이 별도 수정 없이 동작하는 것을 호환성 기준선으로 둔다.
     - 사용 메서드(최소): `getMe`, `getUpdates` 또는 `setWebhook`, `sendMessage`, `editMessageText`, `answerCallbackQuery`, `sendChatAction`(typing).
     - 인라인 키보드: `reply_markup.inline_keyboard`의 `callback_data`를 그대로 사용해 액션 콜백 전달.
   - **확장 인터페이스(SimpleClaw/OpenClaw 호환 브릿지)**: 텔레그램 기본 스펙으로는 표현 어려운 카드·트레이스를 위한 확장 채널.
     - API endpoint: `GET /v1/messages`, `POST /v1/messages`, `POST /v1/webhook/actions`.
     - 인증 방식: HTTPS 기반 `Authorization: Bearer <API_KEY>`.
     - 스트리밍: 응답 청크와 trace 이벤트를 SSE 또는 WebSocket으로 전송.
     - 카드/버튼 메타데이터: 메시지에 첨부되는 JSON으로 카드 타입(`shopping_item`, `trip_plan`, `calendar_event`, `email_summary` 등)과 액션 기술.
     - **Trace 이벤트 스키마**: `event: thinking`(`{step, summary, content}`), `event: tool_call`(`{id, name, args, started_at}`), `event: tool_result`(`{id, status, result_preview, latency_ms}`), `event: message_delta`(텍스트 청크). 텔레그램 호환 모드만 사용하는 봇과는 자동 fallback(트레이스 미표시) 처리.
   - **버디 등록 흐름**: 사용자가 입력한 봇 토큰(또는 `agent://add?token=...` 딥링크)을 검증 → `getMe`로 봇 정보 조회 → 채널(=버디) 레코드 생성 → 토큰은 Secure Storage에 저장.

   5.4 데이터 모델
   - `Buddy` (= `AgentChannel`): 등록된 에이전트 버디. 봇 토큰, 표시명, 아이콘, 색상, 알림 정책, 역할 태그(work/personal 등), 호환 모드(`telegram` / `extended`).
   - `Message`: 개별 대화 내역 로컬 캐시 (텍스트/카드/액션/trace 참조).
   - `Trace`: 메시지에 연결된 thinking·tool_call·tool_result 노드의 시퀀스. 펼침/접힘 상태와 원본 JSON 미리보기 저장.
   - `SavedItem`: 사용자가 아카이브한 링크/요약/메모 (UC-07, UC-08).
   - `OAuthAccount`: 외부 서비스 연동 상태 (만료/스코프).
   - `UserPreferences`: 앱 전역 설정 (테마, 기본 trace 펼침 여부, 방해금지 시간대 등).
   - `LocalSession`: 크래시 복구용 활성 상태 스냅샷.

   5.5 아키텍처 위험
   - 개인정보 처리: 이메일·연락처 등 민감 데이터 흐름 최소화 및 디바이스 내 처리 우선. Trace 패널 인자 JSON에 민감값 마스킹 필수.
   - 다수 버디의 자원 비용: 각 버디별 폴링/웹훅을 동시에 유지할 때 배터리·네트워크 사용량을 최적화(웹훅 우선, 폴링은 fallback).
   - Trace 데이터 폭증: 한 응답이 수십 개 tool call을 발생시킬 수 있어 페이지네이션·요약/펼침 UX 필요.
   - OAuth UX: 모바일에서 외부 서비스 재인증 과정의 매끄러운 처리(딥링크 복귀).
   - 음성 처리 Latency: 로컬 온디바이스 STT 활용 및 스트리밍 파이프라인.
   - 모바일 백그라운드 프로세스: 네이티브 Push 알림(FCM/APNs) 의존, 포그라운드 시 동기화.
   - LLM Rate Limit/응답 Hang: 작업 중 인디케이터 표출 및 타임아웃/재시도 버튼 제공.
   - 텔레그램 호환과 확장 기능의 격차: 표준 봇은 trace·카드 미지원이라 자동 fallback 정책과 사용자 안내 필요.

   - **다중 버디 등록·관리**: 토큰/딥링크 기반 신규 에이전트 추가, 표시명·아이콘·역할·알림 정책 편집, 회사/개인 등 채널 분리 운영.
   - **LLM 친화적 마크다운 렌더링**: 헤더·표·코드·중첩 리스트 등을 모바일 가독성에 맞게 서식화(Telegram의 원문 노출 문제 해소). 카드형 메시지(쇼핑·여행·일정·이메일 요약)도 함께 지원.
   - 음성 입력(STT) 및 텍스트 음성 출력(TTS).
   - OS 네이티브 Push 알림 수신 및 딥링크 진입, 채널별 알림 정책.
   - 인라인 버튼 렌더링 및 콜백 전송 (일정 추가, 답장 발송, 후보 선택 등).
   - 외부 앱·웹의 OS 공유 시트 입력 수신.
   - 단일 사용자 인증, 로컬 대화/아카이브 캐싱.
   - 다중 에이전트 채널 스위칭 및 통합 Inbox(출처 시각 구분).
   - 외부 서비스 OAuth 연동 카드 UX (재인증/스코프 안내).
   - **에이전트 추론 가시화**: thinking 단계·tool/skill call 인자·결과·소요시간을 펼침/접힘 가능한 trace 패널로 노출, 스트리밍 갱신.
   - **iOS·Android 동등 지원**(iOS 16+ / Android 12+).
