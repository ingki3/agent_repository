# Agent Client — PRD

## 1. Vision & Problem
   1.1 Vision
   Agent Client는 1인 운영자가 모바일에서 SimpleClaw, OpenClaw 등 다중 에이전트와 Telegram 호환 UX로 대화하고 작업을 위임함으로써, 데스크톱 없이도 24시간 무결점 운영과 제어를 가능하게 한다.

   1.2 Problem statement
   - 통점 1 (기존 Telegram 봇의 한계): 일반 Telegram 앱은 메시징 중심이라 에이전트 특화 UI(승인 버튼, 폼)와 상태 표시, 컨텍스트 스레딩 등을 제대로 지원하지 못한다.
   - 통점 2 (다중 에이전트 파편화): 여러 에이전트를 각기 다른 봇으로 운영 시, 단일 통합 뷰(Inbox)가 없어 상태 파악과 컨텍스트 스위칭 비용이 크다.
   - 통점 3 (모바일 제어 부재): 기존 웹 콘솔은 데스크톱에 최적화되어, 이동 중 긴급 알림 수신이나 즉각적인 에이전트 관리가 매우 불편하다.
   - 기존 해결책의 한계: 일반 텔레그램은 확장이 막혀 있고, 웹 UI는 모바일 접근성이 떨어짐.

   1.3 Why now
   익숙한 Telegram UX를 차용해 진입 장벽을 낮추면서도, 통합 Inbox와 에이전트 특화 상호작용(승인, 컨텍스트 제어)을 모바일 네이티브로 제공하는 전용 클라이언트가 필수적이다.

## 2. Target Users & Personas
   2.1 페르소나 1~3명
   **페르소나: 1인 운영자 / 인디 해커 (Operator)**
   - 역할/숙련도: 다수의 AI 에이전트를 이용해 프로덕트를 개발/운영하는 고급 사용자.
   - 대상 에이전트: SimpleClaw, OpenClaw (Telegram 호환 브릿지).

   2.2 페르소나별 핵심 작업 흐름
   - 일상 흐름: PC에서는 코딩 등 딥워크, 외부 이동이나 휴식 중에는 모바일로 에이전트 상태를 모니터링.
   - 트리거: 에이전트의 긴급 알림(장애, 배포 승인 등) 수신 시, 혹은 이동 중 떠오른 태스크를 바로 지시할 때.
   - 핵심 작업: 통합 Inbox에서 여러 에이전트의 메시지를 일괄 확인하고, 원터치로 승인(Approve)하며, 텍스트/음성으로 신규 태스크를 위임.
   - 단일 사용자 시점: 1차 PRD 범위에서는 단일 사용자(Single User, 1인 관제)로 한정. 다중 사용자 권한 모델이나 다중 디바이스 동기화는 복잡도를 크게 높이므로 핵심 가치 구현에 집중.

   3.1 Top-level UC 표
   | ID | 한 줄 설명 | 페르소나 | 빈도 | 임팩트 |
   |----|-----------|---------|------|--------|
   | UC-01 | 긴급 시스템 장애 알림 수신 및 즉시 대응 승인 | 운영자 | 주 1-2회 | 高 |
   | UC-02 | 출근 중 음성/텍스트로 모닝 브리핑 (서버 상태, 크론 결과) 확인 | 운영자 | 매일 | 中 |
   | UC-03 | 이동 중 신규 기능 아이디어 메모 및 개발 태스크 초안 위임 | 운영자 | 주 3-4회 | 中 |
   | UC-04 | 배포 파이프라인 진행 상태 모니터링 및 수동 개입 | 운영자 | 주 2-3회 | 高 |
   | UC-05 | 특정 로그/에러 트레이스 조회 요청 | 운영자 | 주 3-4회 | 中 |

   3.2 핵심 UC 3개 상세 (User Journey)
   **UC-01: 긴급 시스템 장애 알림 수신 및 즉시 대응 승인**
   - 트리거: 에이전트(SimpleClaw)가 서버 CPU 스파이크를 감지하고, 인스턴스 재시작 권한을 요청하는 Push 알림 발송.
   - 단계별 흐름:
     1. 시스템: 스마트폰에 Push 알림 전송. "서버 A CPU 95% 지속 중. 자동 재시작을 승인하시겠습니까? [승인] / [거절]"
     2. 사용자: 알림을 탭하여 Agent Client 진입, 상세 컨텍스트(직전 로그 스니펫) 확인.
     3. 사용자: 인라인 [승인] 버튼 클릭.
     4. 시스템: 에이전트에 승인 웹훅/이벤트 전송, 로딩 인디케이터 표시.
     5. 시스템: "재시작 명령 전달됨. 1분 후 상태 브리핑 예정" 메시지 수신.
   - 성공 종료 조건: 에이전트가 재시작을 완료하고 시스템 정상화 알림을 수신함.
   - 실패/대안 경로: 재시작 실패 시, 사용자에게 에러 로그와 함께 수동 SSH 접속 권고 혹은 "추가 진단(Investigate) 요청" 버튼 표시.
   - 사용된 에이전트 capability: Alerting webhook, 승인(Approval) 툴, 인스턴스 관리 스킬.

   **UC-02: 출근 중 모닝 브리핑 확인**
   - 트리거: 출근길에 사용자가 앱을 열어 간밤의 시스템 운영 상태를 묻는다.
   - 단계별 흐름:
     1. 사용자: 하단 마이크 버튼을 눌러 "어젯밤 시스템 이상 없었어?"라고 발화.
     2. 시스템: STT 변환 후 텍스트 메시지 버블로 표시, 에이전트(OpenClaw)로 전송.
     3. 시스템: 에이전트가 "전체 정상입니다. 단, 백업 크론 1개가 실패했습니다."라는 텍스트 응답 및 TTS 오디오 반환.
     4. 사용자: "실패한 크론 다시 돌려줘" 텍스트 입력.
     5. 시스템: 에이전트가 재시도 수행 후 "성공적으로 백업되었습니다" 회신.
   - 성공 종료 조건: 장애 내역 확인 및 후속 조치(재시도) 완료.
   - 실패/대안 경로: 음성 인식 실패 시 텍스트 편집 모드로 전환하여 재입력.
   - 사용된 에이전트 capability: 시스템 모니터링 툴, 크론 관리 스킬, STT/TTS 브릿지.

   **UC-03: 이동 중 아이디어 메모 및 개발 태스크 초안 위임**
   - 트리거: 산책 중 새로운 기능 구현 아이디어가 떠올라 바로 에이전트에게 지시.
   - 단계별 흐름:
     1. 사용자: Agent Client를 열고 "새로운 이슈 추가해줘. 제목은 '다크모드 지원', 내용은 'admin.pen 참조해서 디자인 반영'" 입력.
     2. 시스템: 에이전트(SimpleClaw)가 이슈 트래커 툴을 호출.
     3. 시스템: "[BIZ-180] 이슈가 생성되었습니다. 지금 이어서 디자인 검수를 시작할까요?" 응답.
     4. 사용자: "아니, 나중에" 입력.
   - 성공 종료 조건: 아이디어가 유실되지 않고 시스템의 이슈로 생성됨.
   - 실패/대안 경로: 에이전트가 파라미터(제목, 내용) 부족을 감지하면 추가 질문을 역으로 던져 컨텍스트를 채움.
   - 사용된 에이전트 capability: 이슈 관리 스킬, 텍스트 요약 레시피.

   3.3 Out-of-scope use cases
   - 일반 지인과의 개인 메시징: 에이전트와의 커뮤니케이션에 특화되어 있으며, 일반 P2P 채팅 기능은 제외.
   - 모바일 코드/에디터 직접 수정: 코딩은 PC나 에이전트에 위임하며 클라이언트는 결과물 확인용.
   - 복잡한 차트 및 대시보드 편집: 단순 상태 모니터링은 지원하나 복잡한 커스텀 대시보드는 제외.
   - 다중 사용자 공유 스레드: 여러 사용자가 단일 에이전트와 소통하는 그룹 챗 기능은 1차 제외.

   4.2 Non-functional Requirements
   - 성능: 앱 콜드 스타트는 2초 이내, 로컬 렌더링 지연은 100ms 이내.
   - 보안: 에이전트 연동용 API Key는 OS 네이티브 Secure Storage에 저장, 통신은 HTTPS로 암호화.
   - 접근성: 스크린 리더 지원 및 명도 대비 WCAG AA 기준 충족.
   - 국제화: 한국어 최우선, LTR 방향 적용.
   - 안정성: 네트워크 불안정 감지 및 크래시 복구(직전 채팅방 진입) 지원.
   - 관측성: 크래시 리포트 도구 연동.
   - 호환성: iOS 16+, Android 12+ 지원.

   4.3 Constraints & Assumptions
   - 기술 제약: 1차 릴리스는 단일 기기 기반의 로컬 저장소를 우선 활용한다.
   - 비즈니스 제약: 크로스 플랫폼 프레임워크(React Native, Expo)를 사용한다.
   - 외부 의존성: 에이전트 백엔드 측에 텔레그램 호환 브릿지 혹은 API 레이어가 안정적으로 구현되어 있다고 가정한다.

## 5. Technical Architecture
   5.1 시스템 컨텍스트 다이어그램
   ```
   [User]
     ↓ voice/text (OS Native STT/TTS)
   [Agent Client (Mobile App)]
     ↓ HTTP (REST) / SSE (Streaming) / WebSocket / FCM/APNs (Push)
   [Agent Gateway / Bot API Bridge]
     ↓
   [SimpleClaw / OpenClaw Agent Daemon] ←→ [LLM Routers (Claude/Gemini/OpenAI)]
                                        ←→ [Memory · Skills · Recipes · Cron]
   ```

   5.2 기술 스택 결정
   - 언어 / 프레임워크: React Native (Expo) - 크로스 플랫폼 개발 및 네이티브 모듈 연동.
   - 상태 관리: Zustand - 가볍고 빠른 기능 구현.
   - 음성 입출력: expo-speech (TTS), @react-native-voice/voice (STT).
   - 인증: API Token + Expo SecureStore.
   - 저장: SQLite (expo-sqlite) - 오프라인 캐시 및 컨텍스트 복구.
   - 빌드/배포: EAS (Expo Application Services).

   5.3 SimpleClaw 연동 인터페이스
   - API endpoint: `GET /v1/messages`, `POST /v1/messages`, `POST /v1/webhook/actions`.
   - 인증 방식: HTTPS 기반 `Authorization: Bearer <API_KEY>`.
   - 스트리밍: 에이전트 응답 지연을 낮추기 위해 SSE 또는 WebSocket을 통한 청크 스트리밍 권장.

   5.4 데이터 모델
   - `AgentChannel`: 에이전트 채널 메타데이터
   - `Message`: 개별 대화 내역 로컬 캐시
   - `UserPreferences`: 앱 전역 설정
   - `LocalSession`: 크래시 복구용 활성 상태 스냅샷

   5.5 아키텍처 위험
   - 음성 처리 Latency: 로컬 온디바이스 STT 활용 및 스트리밍 파이프라인.
   - 모바일 백그라운드 프로세스: 네이티브 Push 알림(FCM/APNs)에 의존하고 포그라운드 시 동기화.
   - LLM Rate Limit/응답 Hang: 작업 중 인디케이터 표출 및 타임아웃/재시도 버튼 제공.

## 6. Success Metrics & Scope
   6.1 KPI 표
   | 지표 | 측정 방법 | 목표값 (1차) | 목표값 (6개월) |
   |------|----------|-------------|---------------|
   | 일일 활성 세션 (DAU) | 앱 foreground 진입 횟수 | 3+ (운영자 1인 기준) | 5+ |
   | 알림 대응 속도 | Push 수신 후 인라인 승인 액션 소요 시간 | < 60초 | < 30초 |
   | 음성/텍스트 응답 지연 | 사용자 입력 후 에이전트 1차 응답까지 | < 4s (p50), < 8s (p95) | < 2s (p50) |
   | 액션(승인) 성공률 | 버튼 클릭 대비 정상 처리 응답 비율 | > 95% | > 99% |
   | 무중단 연속 사용 | 강제 종료 없는 연속 세션 수 | > 98% | > 99.5% |

   6.2 Scope In
   - 텔레그램 호환 봇 API 브릿지를 통한 양방향 메시징
   - 텍스트 및 기본 마크다운 렌더링
   - 음성 입력(STT) 및 텍스트 음성 출력(TTS)
   - OS 네이티브 Push 알림 수신
   - 인라인 버튼 렌더링 및 콜백 전송 (승인/거절)
   - 단일 사용자 인증 및 로컬 대화 캐싱
   - 다중 에이전트 채널 스위칭

   6.3 Scope Out
   - 다중 디바이스 동기화
   - 다중 사용자 지원 및 그룹 채팅
   - 이미지/동영상/파일 첨부 전송
   - 복잡한 커스텀 UI 위젯 동적 생성
   - Telegram 일반 사용자(Human)와의 메시징 연동
